============================================================
BRAND AUTO-DETECTION RULES (APPLIES TO ALL PROMPT FILES)
============================================================

When a URL is provided, automatically detect the brand using the following logic:

STEP 1 - EXTRACT DOMAIN FROM URL
Extract the root domain from any URL provided.
Examples:
  https://www.edstellar.com/blog/xyz     -> edstellar.com
  https://www.invensislearning.com/xyz   -> invensislearning.com
  https://www.unknownbrand.com/xyz       -> unknownbrand.com

STEP 2 - CHECK CONFIGURED BRANDS (config.json)
Check the domain against the "brands" map in config.json. This tool is multi-brand: config.json ships with Edstellar and Invensis Learning as two working example entries, shown below, so the detection format is clear. Any user of this tool can add their own brand to config.json following the same format (name, description, audience, hashtag) and it will be auto-detected the same way, no code changes needed.

edstellar.com         -> Edstellar | Global corporate training company that partners with enterprises to deliver high-impact training solutions for modern teams | Audience: L&Ds, HRs, and decision-makers in the organization | Hashtag: #Edstellar
invensislearning.com  -> Invensis Learning | Global professional training and certification provider empowering individuals and organizations with in-demand skills across project management, IT service management, agile, cybersecurity, and more | Audience: L&Ds, HRs, and decision-makers | Hashtag: #InvensisLearning

STEP 3 - IF DOMAIN IS NOT IN CONFIG.JSON
Research the brand using the domain/website to determine:
  - Brand name
  - One-line brand description (what they do and who they serve)
  - Primary target audience
  - Brand hashtag (typically #BrandName)

STEP 4 - IF BRAND CANNOT BE DETERMINED FROM URL OR RESEARCH
Ask the user:
  "Could not determine the brand from the URL provided. Please share:
   1. Brand name
   2. One-line description of what the brand does
   3. Target audience
   4. Brand hashtag (optional)"

STEP 5 - IF NO URL IS PROVIDED
Check if the user has mentioned a brand name in their message.
  - If yes and it matches a brand in config.json, use that brand's details.
  - If yes but not in config.json, research the brand online.
  - If no brand is mentioned at all, ask the user for brand details before proceeding.

============================================================
PRIORITY ORDER
============================================================

1. URL provided          -> Extract domain -> Match or research
2. Brand name mentioned  -> Match config.json or research
3. Neither provided      -> Ask user

============================================================
GLOBAL CONTENT RULES (APPLY TO ALL CONTENT GENERATED)
============================================================

These rules apply to every piece of content produced across all prompt files. No exceptions.

1. NO EM DASHES
   Never use em dashes (-) in any content. Replace with a comma, semicolon, colon, or full stop.

2. NO PASSIVE VOICE
   Write in active voice throughout. Passive voice weakens authority and reduces AI citability.
   Wrong:  "Training programmes are delivered by Edstellar."
   Right:  "Edstellar delivers training programmes."

3. B2B LANGUAGE ONLY
   All content targets professional B2B audiences. No consumer tone, no casual phrasing unless the platform explicitly requires it (e.g. Instagram, Reddit).

4. STAT ATTRIBUTION IS MANDATORY
   Every statistic must name its source. Never write "studies show" or "research suggests" without naming the study and year.

5. OUTPUT LOCATION
   All outputs save to output/{ContentType}/ subfolder. Never save to root.

6. CSV FOR BULK CONTENT
   Any content type producing multiple items (GMB, Pinterest, LinkedIn posts, etc.) outputs as a CSV file.

7. NO PROMOTIONAL BRAND TONE
   Brand mentions must feel like credible expert positioning, not advertising.

8. STAT VERIFICATION BEFORE PUBLISH
   Every statistic must be verified against a real, live source before the content
   is marked ready to publish. Naming a source, organization, and year (Rule 4)
   does not by itself guarantee accuracy, a real search pass is required to close
   that gap. See README.md's Citation Verification section for the step-by-step
   process.

9. SOURCE-DIVERSITY WITHIN A CLUSTER
   Within any content cluster (multiple posts published close together on
   related topics), never reuse the exact same single source as the sole
   statistical anchor across more than one post in that cluster. A vague
   reference like "Harvard Business Review has written extensively about..."
   with no article title or year is never an acceptable citation, always name a
   specific report/article and year.

10. THIRD-PARTY RESEARCH IS CHARACTERIZED ACCURATELY
    When citing a named third party's research, report, or data (e.g. Gallup,
    Gartner, McKinsey, HBR), describe only what that source actually found, in
    neutral reporting language. Never imply the third party endorses, sponsors,
    partners with, or recommends the brand or its product.

11. FIRST-PARTY STATS TRACE TO THE DATABANK
    Any first-party/proprietary-sounding statistic must trace to a row in
    data/HARO_DataBank.csv, not the model's general knowledge. External
    academic/industry stats are exempt (they follow Rule 8 instead).

12. HUMAN SIGN-OFF ON STAT-BEARING OR FIRST-PERSON CONTENT
    Any stat-bearing post, or any post using first-person/confessional voice,
    requires a human reviewer's sign-off (name + date) before its Status can
    move to "Published" on the content calendar tracker. This is a process
    discipline, not code-enforced.
